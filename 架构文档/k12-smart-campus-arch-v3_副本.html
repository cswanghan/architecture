关键架构问题与解决方案
问题 1
解决传统 SaaS 与 AI Agent 双轨并存问题，如何面向Agent开发能力
现状
系统已有 500+ API、70+ 路由、4 条业务线，是一个成熟的传统 SaaS。现在要让 AI Agent 也能操作这套系统。
解决方案
在目前SaaS 提供的带登录校验，带权限约束的Restful API基础上，复用并同时增加建设面向 AI 的 Agent API Gateway 这一层（MCP 或者RestAPI）访问传统 SaaS; 并在这一层的基础上包装技能包。
问题 2
解决 AI 幻觉可能带来的操作偏差
现状
教育数据（学生信息、成绩、课表）高度敏感。AI 直接操作有不可逆风险。
解决方案
  1. 审批引擎: 写操作需要当事人确认或者审核流程（Human-in-the-Loop）
  2. 预演沙箱: 先模拟再执行（Simulate-before-Commit）
  3. 四层审计: 调用→决策→编排→数据，全链路可追溯可回滚
问题 3
解决业务 API 的复杂度，设计面向Agent 友好接入的业务抽象
现状
后端 14 模块、60+ Controller、500+ API。AI Agent 不可能直接面对这种复杂度。
解决方案
微服务中心（7 个能力中心）+ Agent API（82 个高级抽象）+ 三层 API 架构（Query/Operation/Orchestration）

关键架构问题与解决方案
问题 1
从"卖软件"到"卖能力"的商业模式转型
现状
传统教育 SaaS 按模块卖 license，客户定制需求高，交付重。
解决方案
  • 将功能拆解为 API 原语（Resource API）和 业务语义（Domain API）
  • 第三方可以基于 API 自建应用，不再依赖我们的前端
  • 最终形态：Skill Market 让学校按需选择工作流，而非买整套系统
落地难点
Dogfooding 是前提——官方应用自己都不走 Open API 的话，API 质量无法保证。
问题 2
AI Agent 的规模化接入
现状
82 个 MCP Tool 已经跑起来了，但只服务内部 Agent，缺乏开放生态。
解决方案
  • Skill Market: 将 Agent 工作流打包为可分发的 Plugin（Skills + MCP config + hooks）
  • MCP Explorer: 让开发者浏览和测试 Tool
  • Playground: 在线调试 API 调用
问题 3
系统复杂度的降维
现状
500+ API、70+ 路由、14 个后端模块——开发者面对的复杂度太高。
解决方案
两层拆分
  • Domain API: 高级业务语义接口（schedule.solve(), student.enroll()），一个调用完成一个业务任务
  • Resource API: 标准 CRUD 端点（/v1/students, /v1/courses），按资源分组
问题 4
底座能力的教育领域化
现状
通用云基础设施（K8S/MySQL/Redis）没有教育语义，开发者需要从数据库表理解业务。
解决方案
四大领域化原语
  • Storage: 不是 MySQL 表，而是 StudentStore / CourseStore / ScheduleStore 等教育领域对象
  • Compute: 不是通用计算，而是 SchedulingEngine / SelectionEngine / EvaluationEngine 等教育专属引擎
  • Network: API Gateway + MCP Gateway 双入口，EventBus + Webhook 双出口
  • Security: IAM + PolicyEngine + AuditTrail + DataClassification 教育数据安全体系
